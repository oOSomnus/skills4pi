---
name: request-refactor-plan
description: 通过用户访谈创建详细的、包含微小提交的重构计划，然后将其记录到项目 .pi 目录下的文件中。当用户想要规划重构、创建重构 RFC 或将重构拆分为安全的增量步骤时使用。
---

当用户想要创建一个重构请求时，将调用此技能。你应该按照以下步骤进行。如果认为不必要，可以跳过某些步骤。

1. 要求用户详细描述他们想要解决的问题以及任何潜在的解决方案思路。

2. 探索仓库以验证他们的断言并了解代码库的当前状态。

3. 询问他们是否考虑过其他选项，并向他们提出其他选项。

4. 就实现方案对用户进行访谈。要极其详细和彻底。

5. 敲定实现的确切范围。确定你计划更改什么以及不计划更改什么。

6. 查看代码库中该区域的测试覆盖情况。如果测试覆盖不足，询问用户他们的测试计划是什么。

7. 将实现分解为微小提交的计划。记住 Martin Fowler 的建议："让每个重构步骤尽可能小，这样你可以始终看到程序在工作。"

8. 将重构计划写入项目 `.pi/` 目录下的文件。对问题描述使用以下模板：

文件路径：`.pi/refactor-plan-{简短描述}.md`

```markdown
## 问题陈述

开发者面临的问题，从开发者的角度。

## 解决方案

问题的解决方案，从开发者的角度。

## 提交计划

一个详细的实现计划。用通俗语言写下计划，将实现分解为尽可能最小的提交。每个提交应该使代码库保持在可工作的状态。

## 决策文档

已做出的实现决策列表。可以包括：

- 将构建/修改的模块
- 将修改的那些模块的接口
- 来自开发者的技术澄清
- 架构决策
- 模式变更
- API 合约
- 具体交互

不要包含具体的文件路径或代码片段。它们可能很快就会过时。

## 测试决策

已做出的测试决策列表。包括：

- 什么构成一个好的测试的描述（只测试外部行为，不测试实现细节）
- 哪些模块将被测试
- 测试的先前艺术（即代码库中类似类型的测试）

## 超出范围

对此重构超出范围的内容的描述。

## 进一步说明（可选）

关于重构的任何进一步说明。
```
