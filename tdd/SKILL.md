# 测试驱动开发

## 哲学

**核心原则**：测试应通过公共接口验证行为，而不是实现细节。代码可以完全改变，测试不应该变。

**好的测试**是集成风格的：它们通过公共 API 执行真实的代码路径。它们描述系统做什么，而不是它如何做。一个好的测试读起来像一个规范——"用户可以用有效购物车结账"精确地告诉你存在什么能力。这些测试在重构后存活，因为它们不关心内部结构。

**坏的测试**与实现耦合。它们 Mock 内部合作者、测试私有方法或通过外部手段验证（比如直接查询数据库而不是使用接口）。警告信号：你的测试在重构时失败，但行为没有改变。如果你重命名一个内部函数而测试失败，那些测试是在测试实现，而不是行为。

参见 [tests.md](tests.md) 的示例和 [mocking.md](mocking.md) 的 Mock 指南。

## 反模式：水平切片

**不要先写全部测试，然后写全部实现。** 这是"水平切片"——将 RED 视为"写所有测试"，GREEN 视为"写所有代码"。

这会产生**糟糕的测试**：

- 批量编写的测试测试的是想象的行为，而不是实际行为
- 你最终测试的是事物的形态（数据结构、函数签名）而不是面向用户的行为
- 测试变得对真正的变化不敏感——它们在行为破坏时通过，在行为正常时失败
- 你在真正理解实现之前就过早地确定了测试结构

**正确的方法**：通过追踪子弹进行垂直切片。一个测试 → 一个实现 → 重复。每个测试响应你从上一个周期学到的东西。因为你刚刚写了代码，你确切地知道什么行为重要以及如何验证它。

```
错误（水平）：
  RED:   test1, test2, test3, test4, test5
  GREEN: impl1, impl2, impl3, impl4, impl5

正确（垂直）：
  RED→GREEN: test1→impl1
  RED→GREEN: test2→impl2
  RED→GREEN: test3→impl3
  ...
```

## 工作流程

### 1. 规划

在写任何代码之前：

- [ ] 与用户确认需要什么接口变更
- [ ] 与用户确认要测试哪些行为（优先排序）
- [ ] 识别 [深模块](deep-modules.md) 的机会（小接口，深实现）
- [ ] 设计 [可测试性](interface-design.md) 的接口
- [ ] 列出要测试的行为（不是实现步骤）
- [ ] 获得用户对计划的批准

问："公共接口应该是什么样子？哪些行为最重要需要测试？"

**你无法测试一切。** 与用户确认哪些行为最重要。将测试精力集中在关键路径和复杂逻辑上，而不是每个可能的边缘情况。

### 2. 追踪子弹

写一个测试，确认系统的一件事：

```
RED:   为第一个行为编写测试 → 测试失败
GREEN: 编写最小的代码使通过 → 测试通过
```

这是你的追踪子弹——证明路径端到端工作。

### 3. 增量循环

对每个剩余行为：

```
RED:   写下一个测试 → 失败
GREEN: 最少的代码使通过 → 通过
```

规则：

- 一次一个测试
- 只写足够通过当前测试的代码
- 不要预期未来的测试
- 保持测试专注于可观察行为

### 4. 重构

所有测试通过后，寻找 [重构候选](refactoring.md)：

- [ ] 提取重复
- [ ] 深化模块（将复杂性移到简单接口后面）
- [ ] 在自然时应用 SOLID 原则
- [ ] 考虑新代码揭示了现有代码的什么
- [ ] 每个重构步骤后运行测试

**红时决不重构。** 先到达 GREEN。

## 每循环检查清单

```
[ ] 测试描述行为，而不是实现
[ ] 测试仅使用公共接口
[ ] 测试能在内部重构后存活
[ ] 代码对当前测试是最少的
[ ] 没有添加推测性功能
```
