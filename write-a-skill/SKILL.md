# 编写技能

## 流程

1. **收集需求** - 询问用户：
   - 技能覆盖什么任务/领域？
   - 应处理什么具体用例？
   - 需要可执行脚本还是只需要指令？
   - 是否要包含任何参考材料？

2. **起草技能** - 创建：
   - 带有简洁指令的 SKILL.md
   - 如果内容超过 500 行，额外的参考文件
   - 如果需要确定性操作，工具脚本

3. **与用户审查** - 展示草稿并询问：
   - 是否覆盖你的用例？
   - 有什么缺失或不清晰的地方吗？
   - 任何部分需要更多/更少的细节？

## 技能结构

```
skill-name/
├── SKILL.md           # 主要指令（必需）
├── REFERENCE.md       # 详细文档（如需要）
├── EXAMPLES.md        # 使用示例（如需要）
└── scripts/           # 工具脚本（如需要）
    └── helper.js
```

## SKILL.md 模板

```md
---
name: skill-name
description: 能力的简要描述。当 [具体触发条件] 时使用。
---

# 技能名称

## 快速开始

[最小工作示例]

## 工作流程

[复杂任务的带检查清单的逐步流程]

## 高级特性

[链接到单独的文件：参见 [REFERENCE.md](REFERENCE.md)]
```

## 描述要求

描述是**你的代理在选择加载哪个技能时唯一看到的信息**。它在系统提示中与所有其他已安装的技能一起展示。你的代理读取这些描述并基于用户的请求选择相关技能。

**目标**：给代理足够的信息来知道：

1. 此技能提供什么能力
2. 何时/为什么触发它（具体关键词、上下文、文件类型）

**格式**：

- 最多 1024 字符
- 用第三人称编写
- 第一句：它做什么
- 第二句："当 [具体触发条件] 时使用"

**好的示例**：

```
从 PDF 文件中提取文本和表格，填写表单，合并文档。当处理 PDF 文件或用户提到 PDF、表单或文档提取时使用。
```

**坏的示例**：

```
帮助处理文档。
```

坏的示例让代理无法将此技能与其他文档技能区分开来。

## 何时添加脚本

在以下情况下添加工具脚本：

- 操作是确定性的（验证、格式化）
- 相同的代码会重复生成
- 错误需要显式处理

脚本比生成的代码节省 token 并提高可靠性。

## 何时拆分文件

在以下情况下拆分为单独的文件：

- SKILL.md 超过 100 行
- 内容有明确的领域划分（财务 vs 销售模式）
- 高级功能很少需要

## 审查检查清单

起草后，验证：

- [ ] 描述包含触发条件（"当 ... 时使用"）
- [ ] SKILL.md 在 100 行以内
- [ ] 没有时间敏感信息
- [ ] 一致的术语
- [ ] 包含具体示例
- [ ] 引用深度为一层
