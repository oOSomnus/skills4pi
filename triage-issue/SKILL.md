# 分类问题

调查报告的问题，找到其根本原因，并将带有 TDD 修复方案的文件写入项目 `.pi/` 目录下。这是一个大部分不需要干预的工作流——尽量减少向用户提问。

## 流程

### 1. 捕获问题

从用户那里获取问题的简要描述。如果他们还没有提供，问一个问题："你看到的问题是什么？"

不要立即提出后续问题。立即开始调查。

### 2. 探索和诊断

使用 Agent 工具，subagent_type=Explore 深入调查代码库。你的目标是找到：

- **哪里**出现 bug（入口点、UI、API 响应）
- **什么**代码路径涉及（追踪流程）
- **为什么**失败（根本原因，不只是症状）
- **什么**相关代码存在（类似模式、测试、相邻模块）

查看：
- 相关源文件及其依赖
- 现有测试（测试了什么，缺失了什么）
- 受影响文件的最新更改（对相关文件执行 `git log`）
- 代码路径中的错误处理
- 代码库中其他正确工作的类似模式

### 3. 确定修复方法

基于你的调查，确定：

- 修复根本原因所需的最小更改
- 哪些模块/接口受影响
- 需要通过测试验证哪些行为
- 这是回归、缺失功能还是设计缺陷

### 4. 设计 TDD 修复方案

创建一个具体的、有序的 RED-GREEN 循环列表。每个循环是一个垂直切片：

- **RED**：描述一个捕获破坏/缺失行为的具体测试
- **GREEN**：描述使该测试通过的最小代码更改

规则：
- 测试通过公共接口验证行为，而不是实现细节
- 一次一个测试，垂直切片（不是全部测试在前，然后全部代码）
- 每个测试应能在内部重构后存活
- 如果需要，包含最终的重构步骤
- **持久性**：只建议能在激进代码库变更后存活的修复方案。描述行为和契约，而不是内部结构。测试断言可观察结果（API 响应、UI 状态、用户可见效果），而不是内部状态。一个好的建议读起来像一个规范；一个坏的建议读起来像一个 diff。

### 5. 创建修复方案文件

将修复方案写入项目 `.pi/` 目录下的文件中。不要要求用户在创建前审查——直接创建并分享文件路径。

文件路径：`.pi/bugfix-{简短描述}.md`

```markdown
## 问题

对 bug 或问题的清晰描述，包括：
- 发生了什么（实际行为）
- 应该发生什么（期望行为）
- 如何复现（如适用）

## 根本原因分析

描述你在调查中发现的内容：
- 涉及的代码路径
- 为什么当前代码失败
- 任何促成因素

不要包含与当前代码布局耦合的具体文件路径、行号或实现细节。改为描述模块、行为和契约。该文件即使在重大重构后也应保持有用。

## TDD 修复方案

一个编号的 RED-GREEN 循环列表：

1. **RED**：编写一个测试 [描述期望行为]
   **GREEN**：[使其通过的最小更改]

2. **RED**：编写一个测试 [描述下一个行为]
   **GREEN**：[使其通过的最小更改]

...

**REFACTOR**：[所有测试通过后需要的任何清理]

## 验收标准

- [ ] 标准1
- [ ] 标准2
- [ ] 所有新测试通过
- [ ] 现有测试仍然通过
```

创建文件后，打印文件路径和根本原因的一句话摘要。
