# ADR 格式

ADR 存放在 `docs/adr/` 中，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

懒加载创建 `docs/adr/` 目录——只有需要第一个 ADR 时才创建。

## 模板

```md
# {决策的简短标题}

{1-3句话：背景是什么，我们做了什么决定，以及为什么。}
```

就是这样。一个 ADR 可以只是一个段落。价值在于记录 *已经做出了决定* 以及 *为什么* ——而不是填满各个章节。

## 可选章节

只有在能真正增加价值时才包含这些。大多数 ADR 不需要它们。

- **状态** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——当决策被重新审视时有用
- **考虑的选项**——只有被拒绝的替代方案值得记住时
- **后果**——只有需要指出不明显下游影响时

## 编号

扫描 `docs/adr/` 中最高的现有编号，然后加一。

## 何时提供 ADR

以下三个条件必须全部满足：

1. **难以逆转**——以后改变主意的成本是有意义的
2. **没有上下文会令人惊讶**——未来的读者看到代码会想"他们到底为什么这样做？"
3. **真正权衡的结果**——确实存在真正的替代方案，你基于具体原因选择了其中之一

如果决策很容易逆转，跳过——你会直接逆转它。如果它不令人惊讶，没人会好奇为什么。如果没有什么真正的替代方案，没有什么可记录的，除了"我们做了显而易见的事"。

### 什么算合格

- **架构形态。**"我们使用单体仓库。" "写模型是事件溯源的，读模型投射到 Postgres 中。"
- **上下文之间的集成模式。**"订单和账单通过领域事件通信，而不是同步 HTTP。"
- **带有锁定效应的技术选择。**数据库、消息总线、认证提供商、部署目标。不是每个库——只是那些需要花一个季度才能换掉的。
- **边界和范围决策。**"客户数据由客户上下文拥有；其他上下文仅通过 ID 引用。"明确的"不"和有明确的"是"一样有价值。
- **对显而易见路径的有意偏离。**"我们使用手动 SQL 而不是 ORM，因为 X。"任何合理的读者会做相反假设的事情。这些阻止下一个工程师"修复"一个故意的决定。
- **代码中不可见的约束。**"由于合规要求，我们不能使用 AWS。" "响应时间必须在 200ms 以下，因为合作伙伴 API 合约。"
- **当拒绝不明显时的被拒绝替代方案。**如果你考虑了 GraphQL 并因微妙原因选择了 REST，记录下来——否则六个月后又会有人建议使用 GraphQL。
