# 参考

## 依赖类别

在评估深化候选方案时，对依赖进行分类：

### 1. 进程内

纯计算，内存状态，无 I/O。始终可深化——只需合并模块并直接测试。

### 2. 可本地替代

具有本地测试替代的依赖（例如 PGLite 替代 Postgres，内存文件系统）。如果测试替代存在则可深化。深化的模块在测试套件中运行本地替代进行测试。

### 3. 远程但自有（端口和适配器）

跨网络边界的自有服务（微服务、内部 API）。在模块边界定义端口（接口）。深模块拥有逻辑；传输层被注入。测试使用内存适配器。生产使用真实的 HTTP/gRPC/队列适配器。

推荐形态："定义共享接口（端口），为生产实现 HTTP 适配器，为测试实现内存适配器，这样逻辑可以作为一个深模块进行测试，即使它部署时跨越了网络边界。"

### 4. 真正的外部依赖（Mock）

不归你控制的第三方服务（Stripe、Twilio 等）。在边界处 Mock。深化的模块将外部依赖作为注入的端口，测试提供 Mock 实现。

## 测试策略

核心原则：**替代，而不是叠加。**

- 旧的浅模块单元测试在边界测试存在后变成浪费——删除它们
- 在深化模块的接口边界处编写新测试
- 测试断言通过公共接口的可观察结果，而不是内部状态
- 测试应该能在内部重构后存活——它们描述的是行为，而不是实现

## RFC 文件模板

文件路径：`.pi/refactor-rfc-{简短描述}.md`

```markdown
## 问题

描述架构摩擦：

- 哪些模块是浅层的且紧密耦合的
- 它们之间的接缝中存在什么集成风险
- 为什么这使得代码库难以导航和维护

## 提议的接口

选择的接口设计：

- 接口签名（类型、方法、参数）
- 展示调用者如何使用它的使用示例
- 内部隐藏了什么复杂性

## 依赖策略

适用于哪个类别以及如何处理依赖：

- **进程内**：直接合并
- **可本地替代**：使用 [具体的替代] 进行测试
- **端口和适配器**：端口定义、生产适配器、测试适配器
- **Mock**：为外部服务的 Mock 边界

## 测试策略

- **要编写的新边界测试**：描述在接口处要验证的行为
- **要删除的旧测试**：列出变得冗余的浅模块测试
- **测试环境需求**：任何需要的本地替代或适配器

## 实现建议

不与当前文件路径耦合的持久架构指导：

- 模块应该拥有什么（职责）
- 它应该隐藏什么（实现细节）
- 它应该暴露什么（接口契约）
- 调用者应如何迁移到新接口
```
