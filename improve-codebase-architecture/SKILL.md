# 改善代码库架构

像 AI 一样探索代码库，发现架构摩擦，发现提高可测试性的机会，并提出模块深化重构方案作为架构 RFC 文件。

**深模块**（John Ousterhout，《软件设计的哲学》）有一个小型接口，隐藏了大量实现。深模块更可测试、更易于 AI 导航，并且让你可以在边界处测试而不是在内部测试。

## 流程

### 1. 探索代码库

使用 Agent 工具，subagent_type=Explore 自然地导航代码库。不要遵循僵化的启发式规则——有机地探索并记录你遇到摩擦的地方：

- 理解一个概念时，哪些地方需要在许多小文件之间跳转？
- 哪些模块如此浅薄，以至于接口几乎和实现一样复杂？
- 哪些地方纯粹为了可测试性而提取了纯函数，但真正的 bug 隐藏在这些函数被调用的方式中？
- 哪些紧密耦合的模块在它们之间的接缝处产生了集成风险？
- 代码库的哪些部分未经测试或难以测试？

你遇到的摩擦本身就是信号。

### 2. 展示候选方案

展示一个编号的深化机会列表。对每个候选方案，展示：

- **聚类**：涉及哪些模块/概念
- **为什么耦合**：共享类型、调用模式、共同拥有一个概念
- **依赖类别**：参见 [REFERENCE.md](REFERENCE.md) 中的四种类别
- **测试影响**：哪些现有测试会被边界测试替代

不要在此阶段提出接口。问用户："你想探索哪一个？"

### 3. 用户选择候选方案

### 4. 框定问题空间

在生成子代理之前，为所选候选方案编写一个面向用户的问题空间说明：

- 任何新接口需要满足的约束条件
- 它需要依赖的依赖项
- 一个粗略的说明性代码草稿来具体化约束——这不是一个提案，只是将约束落地的办法

将此展示给用户，然后立即进入第 5 步。在子代理并行工作时，用户阅读并思考问题。

### 5. 设计多个接口

使用 Agent 工具并行启动 3 个以上的子代理。每个子代理必须为深化后的模块生成**截然不同**的接口。

为每个子代理提供单独的技术简报（文件路径、耦合细节、依赖类别、隐藏的内容）。此简报独立于第 4 步中面向用户的说明。给每个代理不同的设计约束：

- 代理1："最小化接口——目标是 1-3 个入口点"
- 代理2："最大化灵活性——支持多种用例和扩展"
- 代理3："优化最常见的调用者——使默认情况极其简单"
- 代理4（如适用）："围绕端口和适配器模式设计，用于跨边界依赖"

每个子代理输出：

1. 接口签名（类型、方法、参数）
2. 展示调用者如何使用它的使用示例
3. 内部隐藏了什么复杂性
4. 依赖策略（如何处理依赖——参见 [REFERENCE.md](REFERENCE.md)）
5. 权衡

按顺序展示设计，然后用散文比较它们。

比较之后，给出你自己的推荐：你认为哪个设计最强以及为什么。如果不同设计的元素可以组合得很好，提出一个混合方案。要有主见——用户想要的是明确的判断，而不只是一个菜单。

### 6. 用户选择接口（或接受推荐）

### 7. 创建架构 RFC 文件

将重构 RFC 写入项目 `.pi/` 目录下的文件中。使用 [REFERENCE.md](REFERENCE.md) 中的模板。

文件名格式：`.pi/refactor-rfc-{简短描述}.md`

不需要用户审查——直接创建文件并告知文件路径。
